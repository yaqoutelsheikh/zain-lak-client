# i18n & direction — next-intl

The product ships in four languages and two directions at once. A direction
bug is a layout bug, not a translation bug.

## Mental model

**Locale and direction are data the layout reads — never conditions the
code branches on.** You never write `if (locale === "ar")` to fix a
layout; you write one class that already serves both directions (the
logical-class rule lives in `style.md`). Text is never a literal in the
tree; it is a key resolved from the active locale. Get those two reflexes
right — logical classes, keyed text — and RTL and translation stop being
features you implement and become properties the system already has.

## Structure

- Locale segment `src/app/[locale]/…`; one message file per locale —
  `messages/{en,ar,fr,de}.json`.
- Config in `src/i18n` (`request.ts`); direction/navigation helpers in
  `src/lib/utils/i18n`: `locales`, `getDirection`, `isLocale`, and the
  next-intl `routing` (`routing.defaultLocale`, from `NEXT_PUBLIC_APP_LOCALE`).
- `<html lang={locale} dir={direction}>` is set once, in the root layout,
  from the locale — nowhere else in the tree.
- Locale detection and redirects run as one guard in the proxy chain (see
  `auth.md`).

## Translating — server vs client

Same namespaces, same keys; only the accessor differs.

```tsx
// server component — async
const t = await getTranslations("users");

return <h1>{t("title")}</h1>;
```

```tsx
"use client";

import { useTranslations } from "next-intl";

export function Empty () {

    const t = useTranslations("users");

    return <p>{t("table.empty")}</p>;

}
```

The client tree must sit under `NextIntlClientProvider` (wired in
`[locale]/layout.tsx`), and the server must have called
`setRequestLocale(locale)`. Pass the provider only the namespaces the
client needs — never the whole catalog — so the client bundle stays lean.

## Hard rules

- Zero hardcoded user-facing strings — every visible string goes through
  `t()`. Keys are namespaced by feature: `users.table.empty`.
- A new key lands in BOTH locale files in the same commit. A key present
  in one locale only is a defect.
- Logical classes only (`ps`/`pe`, `ms`/`me`, `start`/`end`, `text-start`,
  `text-end`, `rounded-s`/`rounded-e`) — the physical pairs and the
  `rtl:`/`ltr:` variants are forbidden. The full class rule and its ✅/❌
  live in `style.md`.
- The single `rtl:` exception is a direction-implying icon (arrow,
  chevron) flipped with `rtl:rotate-180` on the icon itself. No other
  `rtl:` exists anywhere.
- Dates, numbers, currencies, and relative time go through next-intl
  formatters (`Intl` underneath). Hand-built locale strings are forbidden.
- Never branch component logic on the locale for layout — if a component
  needs `if (locale === "ar")` to render right, the styling is wrong; fix
  it with logical classes.

## You are doing it wrong if…

- A visible string is a literal in JSX → key it through `t()`.
- You added a key to `en.json` but not `ar.json` (or the reverse) → both,
  same commit.
- You formatted a date or number by hand → use the next-intl formatter.
- You wrote `rtl:` on anything but a flipping icon → it's a logical-class
  job (`style.md`).
- A component reads `locale` to choose a layout → fix it with logical
  classes instead.

- Which Tailwind class to type (logical vs physical), tokens → `style.md`.
- The locale guard in the request chain → `auth.md`.
- Error copy and its keys → `errors.md`.
